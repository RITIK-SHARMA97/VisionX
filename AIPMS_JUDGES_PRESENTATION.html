5-Layer System Architecture

L1: Physical Equipment
Sensors (temp, vibration, pressure, RPM, current, fuel) → IoT Hardware + Python Simulator
L2: IoT Edge
Raspberry Pi → Mosquitto MQTT Broker (QoS 1) → Real-time Data Collection
L3: Data Ingestion & Storage
MQTT Subscriber → Data Quality Validation → SQLite Time-Series DB (Production: InfluxDB)
L4: AI/ML Core
FastAPI Inference Engine (10s cycles) → 3 Independent Models → Alert Generation + Scheduling
L5: Dashboard & Interface
Streamlit Web UI → Real-time Charts, Alerts, Maintenance Schedule, DGMS Reports
🎯 Design Principle: Separation of concerns enables each layer to fail gracefully and be tested independently.

Three Independent AI Models

Model 1: Anomaly Detection (Isolation Forest)

  • Purpose: Classify equipment status → Normal / Warning / Critical (real-time, unsupervised)
  • Input: 42D feature vector (raw sensors + rolling mean/std/slope over 30s window)
  • Output: Anomaly score [0,1] + Label threshold: warning@0.40, critical@0.70
  • Speed: <10ms inference per equipment

Model 2: Failure Prediction (XGBoost Classifier)

  • Purpose: Predict 7-day failure probability with per-feature SHAP explanations
  • Input: 56D tabular features (14 sensors + rolling statistics)
  • Output: P(failure within 7 days) + Top-3 contributing sensors with impact %, expected AUC-ROC > 0.85
  • Speed: <20ms inference per equipment

Model 3: RUL Estimation (LSTM Regression)

  • Purpose: Estimate Remaining Useful Life (days) with confidence intervals
  • Input: 50-step temporal sequences (14 active sensors), zero-padded
  • Output: RUL (hours) + 95% confidence band via bootstrap ensemble
  • Speed: <30ms inference, training < 2 hours on CPU
✓ All 3 models run every 10 seconds independently → if one fails, others continue

Real-Time Data Processing Pipeline

Complete Flow (Sensor to Alert)

1. Sensor Event (Simulator): "EXC-01 vibration = 4.2 mm/s, timestamp 2025-04-15T14:32:01Z"
2. MQTT Transport: Published to topic mines/equipment/EXC-01/sensors (QoS 1)
3. Data Quality Gate: Subscriber validates JSON schema, range checks, flags suspicious readings
4. Time-Series Storage: Write to sensor_readings table with timestamp, value, quality_flag
5. Feature Engineering: APScheduler reads last 5 minutes (300 readings) per equipment every 10s
6. ML Inference: Run all 3 models on normalized, rolling-feature vector
7. Alert Generation: If anomaly_label changes → insert alert record with SHAP top-3 sensors
8. Dashboard Update: Streamlit polls /equipment endpoint every 5s, renders red badge when critical
⏱️ Total End-to-End Latency: <15 seconds from sensor event to engineer alert (target: <10s)

How Failures Develop: The Degradation Curve

Three Lifecycle Phases

Phase 1: HEALTHY (0-60% lifecycle)
Equipment operates within normal parameters. Sensors show: baseline temp ±2%, normal vibration, rated RPM, stable pressure. Random Gaussian noise only.
Phase 2: EARLY DEGRADATION (60-80% lifecycle)
Wear begins - seals loosen, bearings develop play. Sensors show linear drift: temp +5%, vibration +15%, pressure -3%, RPM -2% per hour. SUBTLE but DETECTABLE
Phase 3: ACCELERATED DEGRADATION (80-100% lifecycle)
Failure imminent - wear accelerates exponentially. Sensors spike: temp +20%, vibration +40%, pressure crashes, RPM erratic ±8%. CRITICAL - FAILURE WITHIN HOURS

Real Equipment Example: Excavator Hydraulic Pump

  • Day 1-3: Seal micro-cracks → oil pressure varies ±3% (not obvious to engineer)
  • Day 4-6: Pressure ripple amplitude grows 20%, flow rate drops 8% → EARLY DETECTION WINDOW
  • Day 7: Cavitation begins, pressure cascading failure → 7-day warning expires
  • Day 8: Pump seized, excavator offline, production halted
✓ AIPMS catches Day 4-6 phase → gives engineers 1-3 days to schedule repair before Day 8 failure

Real-Time Sensor Detection: What Sensors "See"

Model 1: Anomaly Detector - Isolation Forest (Unsupervised)

How it works: Learns normal sensor correlations from healthy baseline. Detects when patterns deviate.

Healthy Pattern (99% of data): When pressure rises, temp stable + RPM rises proportionally (linear correlation)
→ Anomaly score = 0.15 (green)
Early Degradation: Pressure rises but temp also rises disproportionately + RPM lags
→ Anomaly score = 0.45 (amber threshold: warning)
Accelerated Degradation: Pressure erratic, temp spikes randomly, RPM drops despite load
→ Anomaly score = 0.82 (red threshold: critical)

Why This Works BEFORE Failure

  • Isolation Forest detects correlation breaking (not just threshold violation)
  • Degradation changes HOW sensors respond to load, not just absolute values
  • Catches Phase 2 (early wear) at 60-80% lifecycle = 7+ days before failure
🔑 Key Insight: Normal equipment: sensors change together. Degrading equipment: sensors change independently → AI sees the pattern break

Model 2: Failure Prediction - XGBoost Classifier

The Question: "Will this equipment fail in next 7 days?"

Training: NASA C-MAPSS turbofan degradation data (100 engines, known failure points)

  • Positive class: Last 30 operating cycles before engine failure = "will fail soon"
  • Negative class: First 60% of lifecycle = "healthy, safe zone"
  • Features: 56D vector: 14 raw sensors + rolling mean/std/slope (captures trend direction)

Real Mining Example: Excavator Gearbox

Time Vibration Oil Temp Pressure Model Output Status
Day 1 2.1 mm/s 45°C 210 bar P(fail) = 8% ✓ Green
Day 4 (6am) 3.8 mm/s ↑ 52°C ↑ 205 bar ↓ P(fail) = 62% ⚠️ Amber Alert

Model 3: RUL Estimation - LSTM Time-Series

The Question: "How many days/hours until this equipment fails?"

LSTM (Recurrent Neural Network): Reads 50-step historical sequences, learns degradation trajectory patterns

How LSTM Sees the Future

  • Input: Last 50 sensor readings (≈ 50 seconds of data at 1 Hz)
  • Processing: LSTM remembers patterns: "when vibration rises like THIS and temp follows THIS shape → failure in ~N days"
  • Output: RUL (Remaining Useful Life) in hours + confidence interval (95% CI band)

Real Example Timeline

Day 1: LSTM predicts RUL = 168 hours (7 days) ± 24 hours [confidence band: 6.8-7.2 days]
Day 4 (6am): LSTM predicts RUL = 96 hours (4 days) ± 18 hours [confidence band: 3.5-4.5 days]
→ Trend line steepening = degradation accelerating
Day 4 (2pm): LSTM predicts RUL = 72 hours (3 days) ± 12 hours [confidence band: 2.8-3.2 days]
→ Critical warning: failure within 72 hours, confidence tightening
Day 7: LSTM predicts RUL = 6 hours ± 2 hours [confidence band: 4-8 hours]
→ Last window before failure, ultra-narrow CI
✓ Asymmetric Loss Function: Penalizes over-predictions 1.5× to keep engineers conservative (safe bias)

The Three-Model Ensemble: Why This Predicts So Well

Model Independence = Robustness

Model Purpose Signal Type Decision
Isolation Forest Anomaly Detection Correlation breaks (pattern-based) Normal/Warning/Critical
XGBoost Failure Risk Historical patterns (supervised) P(fail in 7 days)
LSTM Time Remaining Degradation trajectory (sequential) Days to failure ± CI

Alert Decision Logic (How Engineer Gets Notified)

Green (No Action): Anomaly <0.40 AND P(fail) <0.30 AND RUL >7 days
Amber (Schedule Service): Anomaly 0.40-0.70 OR P(fail) 0.30-0.70 OR RUL 3-7 days
Red (Urgent Service): Anomaly >0.70 OR P(fail) >0.70 OR RUL <3 days

SHAP Explains Every Prediction

When model says "P(fail) = 78%", SHAP shows: "Because vibration trending +40%, pressure down 15%, and RPM inconsistent. These 3 sensors contribute 45% + 33% + 20% of the prediction."

✓ Ensemble Power: If one model errs, other two catch it. No single point of failure.

Why 7-Day Prediction Horizon? The Science

Degradation Follows Exponential Curve

  • Phase 1-2 (0-80% lifecycle): SLOW linear degradation (takes weeks)
  • Phase 3 (80-100%): RAPID exponential degradation (takes 1-3 days)

Detection Timing

  • FMEA Analysis: 7 failure modes have 6-24 hour lead times minimum
  • NASA C-MAPSS Data: Degradation enters "final 30 days" zone predictably
  • Mining Operations: 7-day window = enough time to:
    • Order replacement parts (2-3 days)
    • Schedule technician teams (1 day)
    • Reduce production to alternate equipment (1-2 days)
    • Execute repair (1 day)
    • Test & restart (0.5 days)

Mathematical Basis

Weibull Hazard Function: Mining equipment failures follow Weibull distribution. Phase 3 degradation becomes detectable when equipment reaches ~80% lifecycle. This point statistically occurs 7-10 days before failure in most failure modes.

Production Justification: 7 days is the MINIMUM lead time needed to convert failure from emergency (reactive) to planned maintenance (proactive). Anything less = too little time. Anything more = false positives (wasted technician time).

Technology Stack Rationale

Why These Technologies?

  • Isolation Forest + XGBoost + LSTM: Best balance of accuracy (85%+), explainability (SHAP), and inference speed (<100ms total)
  • FastAPI: Auto-generates OpenAPI docs, async support, 200ms p95 response time on single machine
  • Streamlit: Python-native, no frontend code needed, live rerun every 5 seconds without page refresh
  • SQLite (Prototype) / InfluxDB (Production): Zero-config local dev, purpose-built for time-series in production
  • MQTT: ISO standard IoT protocol, QoS guarantees, proven in mining sites with intermittent connectivity

Production Equivalents (Documented in Design)

  • InfluxDB 3.0 for time-series storage (10-100× query speed vs SQLite)
  • AWS IoT Core for MQTT brokering at scale
  • SageMaker for automated model retraining and A/B deployment
  • Grafana or React frontend for multi-user dashboards
🔄 Clear Path to Production: Every prototype technology has documented enterprise equivalent without architectural rework

Innovation Highlights vs SOTA

Explainability First

  • Every alert includes SHAP top-3 sensor attribution in plain English ("Hydraulic pressure declining -2.1 bar/hr")
